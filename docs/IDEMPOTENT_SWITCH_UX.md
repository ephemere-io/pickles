# スイッチにおける冪等性 ― 物理インターフェースとソフトウェア設計の交差点

## はじめに

冪等性（idempotency）はソフトウェアエンジニアリングにおいて最も重要な設計原則のひとつだ。同じ操作を何度実行しても、結果が変わらないこと。HTTP の PUT は冪等だが POST は冪等ではない。データベースの UPSERT は冪等だが INSERT は冪等ではない。

この概念は、実はソフトウェアの世界に閉じたものではない。我々が毎日触れている物理的なスイッチにも、冪等性の有無がはっきりと存在する。そしてその違いは、UX の良し悪しに直結している。

## 二つのスイッチ

### トグルスイッチ（非冪等）

最も一般的な壁スイッチを考えてほしい。ボタンが一つあり、押すとON、もう一度押すとOFFになる。

```
操作: press()
現在の状態がOFF → ON になる
現在の状態がON  → OFF になる
```

同じ操作 `press()` が、内部状態に依存して異なる結果を生む。これは定義上、非冪等な操作だ。

```
f(OFF) = ON
f(ON)  = OFF
f(f(OFF)) = OFF ≠ f(OFF) = ON
```

つまり `f(f(x)) ≠ f(x)` であり、冪等性を満たさない。

### ポジションスイッチ（冪等）

一方、左右に倒すロッカースイッチ、あるいはスライドスイッチを考える。右に倒せば必ずON、左に倒せば必ずOFF。

```
操作: set_on()  → 常に ON になる（現在の状態に関係なく）
操作: set_off() → 常に OFF になる（現在の状態に関係なく）
```

```
set_on(OFF) = ON
set_on(ON)  = ON
set_on(set_on(OFF)) = ON = set_on(OFF)
```

`f(f(x)) = f(x)` が成立する。冪等だ。

## なぜ冪等なスイッチが優れているのか

### 1. 状態の視認性

冪等なスイッチは、物理的な位置が論理的な状態と一対一に対応する。スイッチを見ただけで、照明がONなのかOFFなのかがわかる。トグルスイッチでは、ボタンの見た目から状態を読み取ることができない。

これは REST API の設計思想と同じだ。リソースの URL を見ればその状態がわかる、という透明性。スイッチの物理的位置がシステムの状態を表現するという透明性。

### 2. 意図の明示性

トグルスイッチで「電気を消したい」とき、まず今の状態を確認しなければならない。ついていれば押す。消えていれば何もしない。操作者は状態の確認と条件分岐を頭の中で行う必要がある。

```python
# トグルスイッチの操作者の思考
if light.is_on():
    press_switch()
else:
    pass  # 何もしない
```

冪等なスイッチでは、意図をそのまま表現できる。

```python
# ポジションスイッチの操作者の思考
switch.set_off()  # 消えてほしい。以上。
```

操作者の認知負荷が圧倒的に低い。

### 3. IoT との親和性

ここが現代において最も重要なポイントだろう。IoT デバイスが物理スイッチと共存する時代において、冪等性は信頼性の基盤になる。

#### ネットワークの不確実性

スマートホームでスイッチの操作をネットワーク経由で送信するとき、パケットロスは日常的に起こる。リトライは必須だ。

```
[スマホ] --"toggle"--> [ロス] --リトライ--> [デバイス]
```

もしトグル操作が2回届いたら、状態が元に戻ってしまう。これは冪等でない操作をリトライした結果だ。分散システムにおける典型的な障害パターンそのものである。

```
[スマホ] --"set_on"--> [ロス] --リトライ--> [デバイス]
```

冪等な操作なら、何度届いても結果は同じだ。`set_on` が3回届こうが10回届こうが、ライトはONになる。それだけだ。

#### 複数コントローラの問題

3路スイッチ（階段の上下にあるスイッチ）の問題を考えてみよう。トグルスイッチで3路回路を組むと、一方のスイッチの物理的位置と照明の状態が乖離する。上のスイッチが「ON位置」にあっても、下のスイッチで消されていれば照明はOFFだ。物理的位置が状態を表現しなくなる。

スマートホームでは、壁スイッチ、スマホアプリ、音声アシスタント、タイマー、モーションセンサーなど、複数のコントローラが一つのデバイスを制御する。トグル型の設計では、どのコントローラから見ても状態が不明になりうる。

冪等な操作（明示的に ON/OFF を指定する操作）であれば、コントローラの数がいくつになろうと、それぞれが意図した状態を正確に指定できる。

#### 状態同期の簡素化

非冪等な操作では、操作を発行する前に現在の状態を問い合わせる必要がある。これは追加のラウンドトリップを要求し、かつ TOCTOU（Time of Check to Time of Use）問題を引き起こす。確認した瞬間と操作した瞬間の間に、別のコントローラが状態を変えるかもしれない。

冪等な操作では、現在の状態を知る必要がない。望む状態を宣言するだけだ。これは宣言的プログラミングの思想そのものであり、Kubernetes のマニフェストが「望む状態」を記述するのと同じ設計思想だ。

### 4. エラーリカバリ

何か問題が起きたとき、冪等な操作は安全に再実行できる。「とりあえずもう一度OFFにする」が副作用なく実行できる。トグルスイッチでは、現在の状態がわからないまま押すのは賭けだ。

## ソフトウェア設計との対応関係

| 物理スイッチ | ソフトウェア | 冪等性 |
|:---|:---|:---:|
| トグルボタン（押すたびにON/OFF） | `POST /light/toggle` | 非冪等 |
| ロッカースイッチ（位置で状態決定） | `PUT /light {"state": "on"}` | 冪等 |
| 押しボタン（押している間だけON） | `POST /light/on` + `DELETE /light/on` | 冪等 |

Pickles プロジェクトとの関連でいえば、分析ジョブの実行において同じ原則が適用される。同じパラメータで同じ分析を再実行した場合、結果は同じであるべきだし、副作用（重複メール送信など）が発生しないよう設計されるべきだ。これが `analysis_runs` テーブルで実行履歴を管理し、冪等なジョブ実行を可能にする設計の根底にある思想と言える。

## 美しいスイッチが存在しない問題

技術的に冪等なスイッチ（ロッカースイッチ、スライドスイッチ）を作ることは難しくない。問題はインダストリアルデザインにある。

現実の市場を見ると：

- **神保電器（JIMBO）**：ミニマルなデザインで及第点。NKシリーズはフラットでクリーンだが、まだ「カッコいい」とまでは言い切れない
- **Panasonic SO STYLE**：かなりいい線をいっている。マットな質感、シンプルな形状。しかし価格が高い
- **海外製品**：Lutron、Legrand などにデザイン性の高い製品はあるが、日本の規格（JIS）との互換性、取付寸法、電気的仕様（100V/日本の配線方式）の壁がある

「四角くしてマットにすれば売れる」という直感は正しいと思う。しかし、スイッチ市場の構造がそれを阻んでいる：

1. **規格の壁**：電気用品安全法（PSE）の認証コスト。新しいデザインのスイッチを一つ出すだけで、相当な認証費用がかかる
2. **量産の経済性**：スイッチのような低単価製品は、金型・製造ラインの初期投資を回収するために大量に売る必要がある。ニッチなデザインスイッチは数が出ない
3. **流通の保守性**：電気工事士が「いつものパナソニック」を指定する慣性。設計事務所経由でないとデザインスイッチは流通しにくい
4. **見えないインフラ**：スイッチは「見えない」前提で設計されてきた歴史がある。壁と一体化して目立たないことが良いとされてきた

IoT の普及により、スイッチは「壁に埋め込まれた電気部品」から「スマートホームのインターフェース」へと役割が変わりつつある。この文脈の変化が、デザインスイッチ市場を拡大させる可能性はある。

## 結論

冪等性は、ソフトウェアでもハードウェアでも、信頼性の高いシステムを設計するための本質的な原則だ。

スイッチにおいて冪等性を実現することは、以下を意味する：

- **操作者の認知負荷を下げる**（現在の状態を知らなくていい）
- **IoT 環境での信頼性を高める**（リトライ安全、複数コントローラ対応）
- **状態の視認性を保証する**（物理位置 = 論理状態）
- **エラーリカバリを簡素化する**（安全に再実行できる）

右に倒したら必ずON、左に倒したら必ずOFF。この単純な原則は、`PUT` の冪等性であり、Kubernetes の宣言的マニフェストであり、関数型プログラミングの参照透過性と、根底では同じ思想を共有している。

あとは、この思想を体現した、カッコいいスイッチが数百円で量産されるのを待つだけだ。
